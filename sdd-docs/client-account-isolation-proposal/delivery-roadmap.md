# 客户公司隔离与账套授权交付 Roadmap

## 1. 文档目标

本文用于将本次需求从“设计方案”收口为“执行路线图”。

本文不讨论候选路线，不列可选项，只定义本次必须完成的交付路径、阶段顺序、并行边界、阶段完成标准和最终上线门槛。

本次 Roadmap 基于以下已确认前提：

- 系统尚未正式上线。
- 当前数据均为测试数据。
- 本次不考虑历史数据迁移。
- 本次必须交付完整闭环，而不是只交付后台配置能力。

## 2. 最终交付目标

本次需求完成后，系统必须达到以下目标：

- 平台超级管理员可以管理客户公司，并可跨客户公司查看和进入财务系统。
- 每个账号只能属于一个客户公司，登录账号全平台唯一。
- 客户公司管理员可以管理本公司的用户、账套和账套授权。
- 普通用户登录后只能看到自己所属客户公司范围内且已授权的账套。
- 不同客户公司之间在用户、账套和财务数据上严格隔离。
- 超级管理员的全局能力通过平台权限实现，不再通过写入账套成员关系伪造。
- 前端、后端、权限、登录和业务进入链路形成完整闭环。

## 3. 交付原则

### 3.1 先立边界，再补功能

客户公司边界、用户归属边界、账套归属边界和接口鉴权边界必须先完成，再开放管理页面和业务入口。

### 3.2 先改后端约束，再改前端体验

前端页面只能建立在后端已形成强约束的前提下。任何只靠页面控制的隔离都不算完成。

### 3.3 先完成管理闭环，再打通业务闭环

必须先打通“客户公司管理 -> 用户管理 -> 账套创建 -> 账套授权”，再打通“用户登录 -> 账套切换 -> 进入做账”。

### 3.4 以单次主线交付为目标

本次交付是一条完整主线，不拆成“先做简版再补完整版”。凡是主线闭环所需能力，全部纳入本次交付。

## 4. 总体阶段划分

本次交付分为五个阶段：

1. 阶段一：建立数据模型和权限基础边界
2. 阶段二：重构后端鉴权与核心接口
3. 阶段三：完成管理端页面闭环
4. 阶段四：完成财务入口与登录后闭环
5. 阶段五：联调、验收与上线准备

阶段一和阶段二构成后端主干，未完成前，不进入前端正式联调。

阶段三和阶段四可以部分并行，但前提是阶段二的接口定义已经稳定。

## 5. 阶段一：建立数据模型和权限基础边界

### 5.1 阶段目标

把“客户公司”从缺失状态变成系统一级实体，并明确用户和账套的归属关系。

### 5.2 必须完成项

- 新增 `wk_client_company` 表。
- 在 `wk_admin_user` 增加 `client_id`、`user_type`、客户管理员标识所需字段。
- 在 `wk_finance_account_set` 增加 `client_id`。
- 为登录账号建立全平台唯一约束。
- 明确平台超级管理员与客户公司用户的身份字段表达。
- 明确账套授权表 `wk_finance_account_user` 的保留策略和同客户约束规则。

### 5.3 本阶段产出

- 数据表结构调整脚本
- 实体类和 Mapper 调整
- 用户身份和客户归属的统一模型
- 可供后续接口复用的客户公司上下文对象

### 5.4 完成标准

- 任一普通用户都能在数据层明确归属到唯一客户公司。
- 任一账套都能在数据层明确归属到唯一客户公司。
- 平台超级管理员不需要也不允许通过账套成员表获得全局权限。
- 数据模型足以支持后续所有接口按客户公司过滤。

## 6. 阶段二：重构后端鉴权与核心接口

### 6.1 阶段目标

让所有管理接口和账套相关接口真正按客户公司边界运行，消除全局视角和越权入口。

### 6.2 必须完成项

- 重构登录后用户信息返回，补齐：
  - `userType`
  - `clientId`
  - `clientName`
  - `isClientAdmin`
- 重构菜单和权限初始化逻辑，去掉“返回全部菜单”的实现。
- 重构财务权限获取逻辑，使其基于当前账套授权和账套角色，而不是全局 finance 菜单切片。
- 重构账套管理接口：
  - 账套列表
  - 新建账套
  - 编辑账套
  - 删除或停用账套
  - 账套授权查询
  - 账套授权保存
- 重构账套切换接口，使其同时校验客户公司边界和账套访问权。
- 补齐客户公司管理接口：
  - 列表
  - 新建
  - 编辑
  - 停用/启用
- 补齐客户公司用户管理接口：
  - 列表
  - 新建
  - 编辑
  - 启用/停用
  - 设置/取消客户管理员
  - 重置密码
- 将所有带 `accountId` 的核心业务接口纳入统一校验：
  - 当前用户身份
  - 当前用户所属客户公司
  - 当前账套所属客户公司
  - 当前用户是否有账套访问权

### 6.3 本阶段产出

- 后端统一权限判定链路
- 客户公司管理 API
- 客户公司用户管理 API
- 账套管理和授权 API 的新实现
- 账套切换和用户登录初始化 API 的新实现

### 6.4 完成标准

- 客户公司管理员调用任何管理接口时，只能作用于本公司资源。
- 普通用户无法通过直接传参访问未授权账套。
- 平台超级管理员可跨公司查看，但不会被写入账套成员关系。
- 前端联调所需的 API 字段已经稳定，不再频繁变更返回结构。

## 7. 阶段三：完成管理端页面闭环

### 7.1 阶段目标

交付平台超级管理员和客户公司管理员的管理端能力，让客户公司、用户、账套和授权形成完整配置链路。

### 7.2 必须完成项

- 新增客户公司管理页：
  - 列表
  - 搜索
  - 新建
  - 编辑
  - 停用/启用
- 新增客户公司用户管理页：
  - 用户列表
  - 新建用户
  - 编辑用户
  - 启用/停用
  - 设置客户管理员
  - 重置密码
- 改造账套管理页：
  - 平台超管先选客户公司再看账套
  - 客户公司管理员默认只看本公司账套
  - 新建账套自动带入当前客户公司
  - 授权弹窗只展示当前客户公司用户
- 改造账套授权选择器：
  - 去掉对全局组织树存根的依赖
  - 改为客户公司内用户列表选择
- 改造管理端路由、菜单和页面权限控制。

### 7.3 本阶段产出

- 平台超级管理员管理端闭环
- 客户公司管理员管理端闭环
- 账套授权页面的新数据源和新交互

### 7.4 完成标准

- 平台超级管理员可以从页面完成客户公司创建、用户管理、账套查看。
- 客户公司管理员可以从页面完成本公司用户和账套授权管理。
- 账套授权时无法看到其他客户公司的用户。
- 前端管理端不再依赖后端兼容存根组织树来完成本次主线功能。

## 8. 阶段四：完成财务入口与登录后闭环

### 8.1 阶段目标

让用户从登录开始，到进入正确客户公司和正确账套做账，全链路按新权限模型运行。

### 8.2 必须完成项

- 改造登录后初始化流程：
  - 获取用户身份
  - 获取客户公司上下文
  - 获取可访问账套集合
- 改造财务主布局 `FmLayout.vue`：
  - 平台超级管理员先切客户公司
  - 再切该客户公司内账套
  - 客户公司用户不展示客户公司切换
- 改造账套默认进入规则：
  - 有已授权账套时进入默认账套
  - 无账套时进入明确空状态页或阻断页
- 改造无账套授权提示：
  - 不能只弹简单提示
  - 必须给出明确状态说明
- 改造账套初始化和跳转链路：
  - 兼容新的客户公司上下文
  - 兼容新的账套访问权校验
- 完成用户密码闭环：
  - 管理员设初始密码
  - 管理员重置密码
  - 用户登录后自行修改密码

### 8.3 本阶段产出

- 登录后身份初始化新链路
- 财务入口两层切换体验
- 无账套空状态和进入规则
- 用户从被创建到可正常做账的完整链路

### 8.4 完成标准

- 新创建用户完成授权后可直接登录并进入被授权账套。
- 无授权用户登录后不能误入做账界面。
- 平台超级管理员在财务端先选客户公司，再在该客户公司内切账套。
- 客户公司普通用户只看到自己有权进入的账套。

## 9. 阶段五：联调、验收与上线准备

### 9.1 阶段目标

验证多客户隔离、账套授权、登录闭环和超级管理员能力已真正成立，而不是局部页面可用。

### 9.2 必须完成项

- 进行后端接口级越权测试。
- 进行前后端联调测试。
- 进行三类角色的端到端验收：
  - 平台超级管理员
  - 客户公司管理员
  - 普通用户
- 进行多客户公司交叉隔离测试。
- 进行无账套、停用用户、停用客户公司等异常场景测试。
- 清理旧的超管写入账套成员逻辑和相关前端兼容逻辑。
- 更新部署说明和测试验收说明。

### 9.3 本阶段产出

- 测试用例清单
- 联调缺陷清单和修复记录
- 最终验收报告

### 9.4 完成标准

- 任意客户公司用户都无法看到其他客户公司的用户、账套和财务数据。
- 任意未授权用户都无法进入未授权账套。
- 平台超级管理员具备全局能力且不污染账套授权数据。
- 所有主线页面和主线接口都已切到新模型运行。

## 10. 并行开发边界

## 10.1 可以并行的工作

- 阶段一完成数据模型定义后，后端可以开始接口改造，前端可以开始静态页面搭建。
- 阶段二接口定义稳定后，前端可以并行开发：
  - 客户公司管理页
  - 用户管理页
  - 账套管理页改造
  - 财务主布局改造

## 10.2 不能并行的工作

- 在后端未建立客户公司边界前，不能开始正式联调账套授权。
- 在后端未稳定登录态返回结构前，不能冻结前端登录后初始化逻辑。
- 在账套切换接口未完成新鉴权前，不能验收财务主布局改造。

## 11. 建议执行顺序

按交付效率和风险控制，执行顺序如下：

1. 完成数据结构和身份模型改造。
2. 完成登录态、权限和核心鉴权链路改造。
3. 完成客户公司管理和用户管理接口。
4. 完成账套管理、账套授权和账套切换接口。
5. 完成管理端页面改造。
6. 完成财务入口、账套切换和无账套空状态改造。
7. 完成联调、越权测试和验收。

这个顺序不能倒置。因为本次需求的核心不是“页面补齐”，而是“边界正确”。

## 12. 角色分工建议

### 12.1 后端主线

负责：

- 数据模型改造
- 权限链路改造
- 登录态改造
- 客户公司、用户、账套和授权接口
- 账套访问权统一校验

### 12.2 前端主线

负责：

- 管理端新增页面
- 账套管理页改造
- 财务主布局改造
- 登录后初始化链路适配
- 无账套空状态和权限引导

### 12.3 测试主线

负责：

- 多角色测试
- 多客户隔离测试
- 越权测试
- 主线闭环测试

## 13. 里程碑定义

### 里程碑 M1：边界建成

判定标准：

- 数据模型完成
- 用户和账套均已带客户公司归属
- 后端已有统一身份和客户上下文

### 里程碑 M2：后端可联调

判定标准：

- 客户公司管理 API 可用
- 用户管理 API 可用
- 账套管理和授权 API 可用
- 账套切换和登录态接口可用

### 里程碑 M3：管理端闭环完成

判定标准：

- 平台超级管理员可完成客户公司管理
- 客户公司管理员可完成用户和账套授权管理

### 里程碑 M4：业务闭环完成

判定标准：

- 新用户创建、授权、登录、进入账套全链路打通
- 无授权用户有明确阻断和提示

### 里程碑 M5：可上线验收

判定标准：

- 多客户隔离测试通过
- 三类角色主线验收通过
- 旧逻辑残留清理完成

## 14. 最终上线门槛

以下条件必须同时满足，才允许视为本次需求完成：

- 数据模型已经支持客户公司一级隔离。
- 后端所有核心接口已经按客户公司和账套权限强校验。
- 客户公司管理、用户管理、账套管理、账套授权均已具备正式页面。
- 登录后初始化和财务入口已切到新模型。
- 平台超级管理员两层切换体验已完成。
- 无账套授权场景已正确阻断。
- 不再依赖组织树兼容存根完成本次主线能力。
- 不再通过超管写入账套成员表实现全局权限。

达到以上门槛后，本次需求才算真正交付。
