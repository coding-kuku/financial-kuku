# 客户公司账套隔离与授权需求 Proposal

## 1. 背景

当前系统最初更偏向单组织内的财务账套管理模式，现在需要对外给多家财务公司客户使用。因此系统需要从“单组织下的账套授权”升级为“多客户公司隔离 + 客户内多账套授权”的模式。

本次需求的核心目标是：

- 支持多家客户公司同时使用同一套系统。
- 不同客户公司之间的数据必须严格隔离。
- 每家客户公司可以创建和管理多个账套。
- 每家客户公司内部管理员可以按账套给内部用户授权。
- 平台超级管理员拥有全局可见和全局管理能力。

## 2. 本次已确认的需求

### 2.1 账号归属规则

- 一个账号只能属于一个客户公司。
- 不允许一个账号同时属于多个客户公司。
- 登录账号应在全平台范围内唯一，避免同名账号跨客户公司造成身份歧义。

### 2.2 客户公司与账套关系

- 一个客户公司可以有多个账套。
- 客户公司内部用户可以被授权到多个账套中做账。
- 是否能访问某个账套，不是按“客户公司内默认全员可见”处理，而是继续采用账套授权机制。

### 2.3 客户公司内部权限模型

- 每个客户公司内部需要有管理员角色。
- 每个客户公司允许存在多个管理员。
- 客户公司管理员可以创建本公司账套。
- 客户公司管理员可以给本公司内部用户分配账套权限。
- 普通用户只能访问自己被授权的账套。

### 2.4 客户公司管理与用户管理

- 平台超级管理员需要有客户公司管理页面。
- 本次需求需要包含完整的客户公司用户管理能力，而不是只做账套授权。
- 客户公司管理员可以管理本公司的用户。
- 用户管理本次至少需要支持：
  - 创建用户
  - 编辑用户
  - 停用/启用用户
  - 设置是否为客户公司管理员
  - 重置密码

### 2.5 客户公司信息维护边界

- 平台超级管理员可以维护客户公司的全部主信息。
- 客户公司管理员可维护本公司的联系人、手机号、备注等运营信息。
- 客户公司名称这类主信息默认由平台超级管理员维护，不由客户公司管理员直接修改。

### 2.6 客户公司之间的隔离要求

- 不同客户公司之间绝对不能相互看到对方的账套信息。
- 不同客户公司之间绝对不能互相看到对方的用户信息。
- 不同客户公司之间绝对不能互相访问对方账套中的做账数据、凭证、报表、辅助核算等业务数据。

### 2.7 平台超级管理员能力

- 平台超级管理员可以看到所有客户公司。
- 平台超级管理员可以看到所有客户公司的账套。
- 平台超级管理员拥有完整管理权限。
- 平台超级管理员的全局能力应通过平台权限实现，而不是通过把超管伪装成每个账套成员来实现。

### 2.8 前端交互层级

- 平台超级管理员在财务管理场景中应采用两层视图：
  - 先切换到某个客户公司
  - 再在该客户公司内切换账套
- 客户公司管理员进入管理端后，只看到自己公司的用户和账套。
- 普通用户进入系统后，只看到自己被授权的账套，不进入客户公司管理视图。
- 当前 `/#/manage/finance/handle` 页面的“账套管理 / 授权账套”交互可以沿用，但数据源和权限边界必须重做。

### 2.9 登录闭环与密码规则

- 本次需求必须做到完整闭环，而不是只做后台配置页面。
- 闭环范围包括：
  - 客户公司创建
  - 用户创建
  - 用户初始密码设置
  - 用户账套授权
  - 用户登录
  - 登录后按权限进入系统
  - 无账套授权时的明确提示
- 本次采用账号密码模式，不依赖手机验证码。
- 创建用户时由管理员设置初始密码。
- 管理员支持重置密码。
- 用户支持登录后自行修改密码。
- 本次不纳入任何依赖手机验证码提供商的需求。

### 2.10 组织架构与部门树范围

- 现有本地后端并没有可直接复用的完整组织架构和部门树管理能力。
- 因此本次需求不以“部门树授权”作为主方案。
- 本次按“客户公司 + 用户列表 + 账套授权”实现闭环。
- 部门树/组织架构能力如未来需要，可在后续阶段单独建设。

## 3. 业务模型

本次需求采用三层模型：

### 3.1 平台层

- 平台超级管理员
- 客户公司管理员

平台层负责决定谁可以管理客户公司、查看全局、创建客户公司、进行跨客户查看。

### 3.2 客户公司层

- 用户必须归属于唯一一个客户公司。
- 账套必须归属于唯一一个客户公司。

客户公司层负责建立最外层的数据隔离边界。

### 3.3 账套层

- 一个账套归属于一个客户公司。
- 一个用户是否可进入某账套，由账套授权关系决定。
- 一个用户可以被授权到本客户公司下的多个账套。

## 4. 现有交互是否可沿用

可以沿用当前 `/#/manage/finance/handle` 页面的“账套管理 / 授权账套”交互思路。

沿用方式如下：

- 保留账套列表。
- 保留“新建账套”能力。
- 保留“授权”操作。
- 保留“一个用户可拥有多个账套权限”的模式。

但必须增加一个前提：

- 页面中的账套列表只能是当前客户公司的账套。
- 页面中的可授权用户只能是当前客户公司内部用户。
- 当前登录用户必须先具备客户公司管理员权限，才能执行本页管理操作。

同时，本次前端不只需要改造账套页，还需要补齐以下完整交互：

- 平台超级管理员的客户公司管理页面
- 客户公司管理员的用户管理页面
- 登录后首页及默认进入路径
- 无账套授权时的提示和引导
- 平台超管“先选客户公司，再选账套”的两层切换体验

也就是说，这个页面的产品形态可以保留，但底层数据边界和权限校验必须重做。

## 5. 当前系统与目标需求的关键差距

### 5.1 缺少客户公司这一层模型

当前系统里账套和用户关系主要围绕账套本身和账套成员展开，但没有明确的“客户公司”边界。这样无法从数据模型上保证客户之间的绝对隔离。

### 5.2 账套可见性仍偏全局视角

当前账套管理相关逻辑更接近全局账套管理，而不是“当前客户公司内账套管理”。这与多客户公司隔离目标不一致。

### 5.3 账套授权仍未天然限定在客户公司内部

虽然已有账套授权能力，但如果底层用户池和账套池没有客户公司边界，那么授权对象和被授权账套仍然可能处于全局视角，存在越界风险。

### 5.4 超级管理员实现方式不适合目标场景

平台超级管理员应该是平台层权限身份，不应通过自动加入每个账套成员表来实现全局可见。否则会污染业务数据，也会让权限模型混乱。

### 5.5 接口层仍需要强制归属校验

即使前端页面做了限制，后端接口仍必须校验：

- 当前用户属于哪个客户公司。
- 当前账套属于哪个客户公司。
- 当前用户是否有权访问或管理该账套。

否则仍然可能出现越权读取、越权授权、越权切换账套的问题。

## 6. 本次需求的范围定义

### 6.1 本次需求范围内

- 增加客户公司边界。
- 保证账号唯一归属客户公司。
- 保证登录账号全平台唯一。
- 保证账套唯一归属客户公司。
- 保留并改造账套授权机制。
- 增加客户公司管理能力。
- 增加客户公司用户管理能力。
- 支持客户公司管理员管理本公司账套和本公司用户授权。
- 支持平台超级管理员全局查看和管理。
- 支持用户从创建、授权、登录到进入账套的完整使用闭环。
- 支持基于账号密码的用户体系。

### 6.2 本次需求范围外

- 一个账号归属多个客户公司。
- 跨客户公司共享账套。
- 跨客户公司共享用户。
- 客户公司之间任何形式的数据互通。
- 本次不做依赖短信验证码提供商的登录、找回密码或校验能力。
- 本次不以部门树/组织架构授权为交付目标。

## 7. 目标权限规则

### 7.1 平台超级管理员

- 可查看全部客户公司。
- 可查看全部账套。
- 可查看全部用户。
- 可管理全部客户公司、账套、授权关系。

### 7.2 客户公司管理员

- 只能查看自己所属客户公司。
- 只能查看自己客户公司的账套。
- 只能查看自己客户公司的用户。
- 可以创建和维护自己客户公司的用户。
- 可以创建自己客户公司的账套。
- 可以给自己客户公司的用户分配账套权限。
- 可以维护本公司的联系人、手机号、备注等运营信息。

### 7.3 普通用户

- 只能查看自己所属客户公司范围内且已授权的账套。
- 只能进入自己已授权的账套做账。
- 无法查看未授权账套。
- 无法查看其他客户公司的任何信息。

## 8. 验收标准

本次需求完成后，应满足以下验收标准：

1. 新建用户时必须归属某一个客户公司，且不能归属多个客户公司。
2. 登录账号在全平台范围内唯一。
3. 新建账套时必须归属某一个客户公司。
4. 平台超级管理员可以创建、编辑、停用客户公司。
5. 客户公司管理员登录后，只能看到本公司的账套和本公司的用户。
6. 客户公司管理员在账套授权页面中，只能给本公司用户授权。
7. 客户公司管理员可以创建本公司用户，并设置初始密码。
8. 普通用户登录后，只能看到自己被授权的账套。
9. 如果普通用户没有任何账套授权，系统应给出明确提示，而不是看到其他客户公司或其他账套信息。
10. 不同客户公司之间不能看到彼此的账套、用户、做账数据。
11. 平台超级管理员可以查看和管理所有客户公司及所有账套。
12. 平台超级管理员在财务视图中应先切换客户公司，再切换该公司的账套。
13. 本次交付不依赖手机验证码能力，账号密码链路可独立闭环运行。

## 9. 总结

本次需求不是简单补几个权限点，而是要把当前系统升级为“客户公司隔离 + 客户内账套授权”的模型。

结论如下：

- `/#/manage/finance/handle` 页面的交互思路可以沿用。
- 但平台超级管理员视图不能直接等同于客户公司管理员视图，需要增加客户公司层级。
- 账套授权机制也可以沿用。
- 本次必须同时补齐客户公司管理、用户管理、登录使用闭环。
- 但必须先补上“客户公司”这一层数据边界与权限边界。
- 后续所有账套可见性、用户可见性、授权操作、账套切换和业务数据访问，都必须建立在客户公司隔离的前提之上。
- 本次先按“客户公司 + 用户列表 + 账套授权”建设，不把部门树和短信验证码能力强行并入主线。
